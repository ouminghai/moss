# One protocol per package, assembled by manifests into empty registries

Protocol adapters live in per-protocol npm packages (`packages/protocols/<name>` → `@mossxyz/protocol-<name>`), not in one shared package. Each package's index exports a **manifest** (`defineProtocolPackage`): the protocols it ships and the tokens it introduces (`tokens.ts` — receipt tokens, LP tokens, LSTs). A `Registry` starts EMPTY; assembly is explicit — `registry.use(manifest)` — and nothing registers itself by import side effects. There is no aggregate meta-package: the MCP server (the batteries-included product surface) assembles its served catalog itself — adding a protocol is one dependency plus one entry in the `use()` array in `server.ts`.

## The three foundation layers

Below the protocol packages sit three layers with one-sentence charters, dependencies pointing strictly downward:

- **`@mossxyz/core` — the machinery.** Decorators, Handles, Plans, Registry, semantic types, the Token identity class, the TokenTable mechanism, the @Event observation authoring surface, the domain types. Zero chain data, zero ABIs, zero defaults: `createRuntime` takes chain identity explicitly, and core reads no contracts — resolving an unregistered token address requires an injected `tokenFallback` (`RegistryOptions`).
- **`@mossxyz/simulator` — the verification engine.** `debug_traceCall` dual-tracer simulation, effects extraction, expects reconciliation, observation wiring. Depends only on core; sits beside erc in the layering. Consumed by the MCP server, by wallets embedding verification, and by every protocol package's mandatory zero-warning e2e tests.
- **`@mossxyz/erc` — the interface layer (ADR 0009).** Compiled interface ABIs (IERC20, IERC721, IWETH9 — forge + @wagmi/cli; public names `ERC20Abi`/`ERC721Abi`/`WETH9Abi`) and address-free generic behavior built on them: the generic `erc20` and `erc721` protocols (`contracts: {}` + injected `runtime`), `approveStep` (approving IS an ERC-20 operation; core owns only the `approval`-tag mechanics that plan() folds into expects), and `erc20MetadataSource` (the standard token-metadata reader, wired into registries as the table fallback).
- **`@mossxyz/system` — the Monad instances.** The well-known token data (`MONAD_TOKENS`), the chain defaults (`MONAD_CHAIN_ID`, `DEFAULT_RPC_URL`, `monadRuntime()`), and the address-bearing system adapters (WMON: the interface is erc's generic WETH9 standard, the address is Monad data). Its `systemManifest` is opt-in like every other package.

The placement rule of thumb: **anything with a hardcoded address lives in `system` or a protocol package — never below.** Deliberate cosmetic exceptions, recorded here so nobody "fixes" them: `Token.native()`'s display symbol "MON", the native symbol in the generic erc20 protocol's balance output, and the endpoint suggestions in the simulator's error text — parameterizing display strings costs more than it buys in single-chain v1.

## Why

- **Contribution unit = ownership unit.** A contributor owns a whole package: independent versioning, per-package bounties, `dependencies` that make composition explicit (Kuru depends on `erc` for the approve encoding and on `system` for canonical token addresses — visible in its package.json, not hidden in imports).
- **The token table is a security surface, so its assembly must be deterministic.** Import-side-effect registration (a global mutable table) would let any transitively-imported package silently remap "USDC"; import order would decide the winner. With registry-scoped assembly, whoever assembles decides, and collisions are hard errors: same symbol → different address rejects the whole manifest (accident or attack, never fine); identical re-registration is idempotent; symbol uniqueness is case-insensitive.
- **No cycles by construction.** An auto-registering core would need to depend on the packages it registers. Uniform manifests make package boundaries pure taste.

## Considered Options

- **Merging the MCP server into core** — structurally impossible, not just distasteful: the server's value is being batteries-included (`npx moss-mcp` boots with the served catalog), so it must depend on every protocol package it serves, which depend on core. The server is the TOP of the dependency chain; core is the BOTTOM. Folding top into bottom is a dependency cycle, and the escape hatches (a server with no default protocols, or config-file dynamic imports) destroy the npx experience. Merging would also put `@modelcontextprotocol/sdk` + zod into core's dependency tree — and therefore into every adapter's and every embedder's.
- **An aggregate bundle package (`@mossxyz/protocols`)** — an earlier revision shipped one; superseded. Its entire content was a manifest array and a few re-exports: a whole publishable package to version, link, and keep in lockstep for one line of code, and its only real consumer (the server) is itself the natural assembly point. Library embedders compose manifests directly — which is also the posture we want them in: choosing what they trust, not inheriting a default catalog.
- **The simulator inside core or inside the MCP server** — it lives in neither: standalone `@mossxyz/simulator`, depending only on core. Not in the server, by dependency direction: protocol packages sit BELOW the server yet need `createTraceSimulator` for their mandatory zero-warning e2e tests (and wallets embed simulation without speaking MCP at all). Not in core: verification is an engine with its own business logic (tracer orchestration, effects vocabulary, reconciliation policy), while core is the authoring substrate every adapter imports — adapters need decorators and Plans, not a tracer. The `expects` schema (the contract) stays in core with Plan; the simulator ships its verifier. The server owns no mechanisms; it is a protocol skin over Registry + Simulator.
- **An `apps/` directory for the server** — the server is a published npm package with a library export (`createMossServer`, for embedding Moss's tools into a host MCP server); `packages/` is where published things live. `apps/` can be introduced when something deploy-only and never-imported exists (landing page, demo UI).
- **System data inside core** (two earlier revisions lived this way) — superseded: it left core owning Monad data and a hand-written ERC-20 ABI. The three-layer split gets core to zero chain knowledge and lets the WETH9/WMON interface come from compiled source like everything else.
- **erc below core** (core importing erc's compiled ABIs) — rejected in favor of moving core's last ABI uses (approve encode, metadata reads) UP into erc where they semantically belong; core at the bottom keeps domain types flowing one way without structural-typing hacks.

The layering each boundary buys: core stays MCP-free and chain-free (viem is its only dependency), the simulator stays embeddable without MCP, the standards stay address-free, the server stays batteries-included.

## Consequences

- `new Registry(runtime)` knows nothing — tests and embedders opt into exactly the packages they want; batteries-included assembly wires `tokenFallback: erc20MetadataSource(client)`.
- Core's tests know no real chain data — fixtures only. Live e2e tests (which inherently touch mainnet) carry their endpoint config as explicit test data.
- New-package scaffolding friction is absorbed by `packages/protocols/_template`: a real, private, CI-built workspace package that is copied (`cp -r`), so the template can never rot.
- Protocols that operate on caller-supplied addresses (generic erc20) declare `contracts: {}` and build Handles from the always-injected `runtime`.
