# Moss 开源贡献计划

## 1. 我的 Builder 身份

我的身份是 **Dev Builder**。我希望通过真实开源项目贡献，学习如何阅读项目架构、理解 Adapter 的设计方式，并最终完成一次可以提交 Pull Request 的功能开发。

本次我选择的贡献方向是：

> 新增一个 NFT mint Protocol Adapter。

## 2. 选择的贡献方向

我计划为 Moss 开发一个新的 NFT mint 协议适配器，让 AI Agent 能够通过 Moss 支持某个 Monad 链上 NFT 项目的公开 mint 能力。

理想目标是接入一个简单的 ERC-721 NFT mint 合约，支持：

- 发现 mint 能力
- 加载 mint 参数说明
- 构建 NFT mint Plan
- 模拟 mint 结果
- 检查用户是否收到 NFT
- 在模拟无 warning 后，作为安全的未签名交易交给钱包查看

对应 Moss 标准流程是：

```text
discover → load → action → simulate
```

## 3. 为什么选择 NFT mint Adapter

我选择 NFT mint Adapter，主要有四个原因。

第一，NFT mint 的链上流程比较直观：

```text
用户支付 MON → 调用 mint → 用户收到 NFT
```

相比 DEX、Lending、Bridge，NFT mint 不需要处理复杂的路由、滑点、借贷健康度、跨链消息等问题，更适合我作为第一次 Protocol Adapter 贡献。

第二，它能覆盖 Moss Adapter 的核心能力。

虽然 NFT mint 比 DeFi 协议简单，但它依然可以完整练习 Moss 的关键设计：

- `@Protocol`
- `@Capability`
- `verb: "mint"`
- `category: "nft"`
- `plan()`
- `expects.out`
- `expects.nfts`
- `@Event` observation
- live e2e simulate

第三，它更容易调试。

NFT mint 的模拟结果比较容易判断：如果成功，应该看到用户地址收到 NFT；如果失败，通常是 mint 关闭、价格错误、白名单限制或参数错误。

第四，它适合作为一个清晰的 GitHub Portfolio。

一个完成度高的 NFT Adapter PR 可以展示我理解了：

- Moss 的 Adapter 架构
- AI Agent 如何扩展链上能力
- 开源项目的代码规范
- 从调研、实现、测试到提交 PR 的完整流程

## 4. 准备完成的任务

### 4.1 协议调研

我会先寻找 2 到 3 个 Monad 上的 NFT 项目，筛选出最适合接入的目标合约。

筛选标准：

- 合约是 ERC-721，暂时不选择 ERC-1155
- mint 仍然开放
- mint 函数参数简单
- 不需要白名单、签名或 Merkle proof
- mint 价格固定或可以从链上读取
- ABI 来源可靠，例如官方仓库、SDK、npm 包或 verified explorer
- mint 后会触发标准 `Transfer` 事件或项目自己的 mint 事件

### 4.2 Adapter 开发

确定目标合约后，我会从模板开始创建协议包：

```bash
cp -r packages/protocols/_template packages/protocols/<nft-project>
```

计划完成：

- 修改 `package.json`
- 添加 ABI 文件和 ABI origin header
- 实现 `src/adapter.ts`
- 实现 `mint` capability
- 根据需要实现 `price` 或 `mintInfo` query
- 在 `src/index.ts` 导出 manifest
- 如果协议引入新 token，在 `src/tokens.ts` 中登记

### 4.3 Plan 设计

mint capability 的 Plan 需要声明清楚实际风险和预期结果。

初步设计：

```text
verb: mint
category: nft
risk: fundOut
```

预期效果：

- `expects.out`：用户最多支付多少 MON
- `expects.nfts`：用户应该收到多少个 NFT

如果合约 mint 是免费的，则 `expects.out` 可以为空或为 0，重点检查 `expects.nfts`。

### 4.4 Observation 设计

我会优先使用标准 ERC-721 `Transfer` 事件作为 mint 的链上证据。

如果目标 NFT 合约有自己的 `Minted` 事件，也会考虑添加 `@Event`，让模拟结果能生成更容易理解的人类可读回执，例如：

```text
Minted 1 NFT from <collection> to <account>
```

同时在 capability 中使用 `confirms`，确保模拟时必须出现对应回执。

### 4.5 测试和验证

我会编写两类测试：

1. 离线测试

   验证：

   - `discover` 能发现 mint 能力
   - `load` 返回正确参数说明
   - `action` 能构建 Plan
   - Plan 中的 `expects` 正确

2. 主网模拟 e2e

   使用 Monad 主网真实状态模拟 mint：

   ```bash
   MOSS_RPC_URL=https://rpc.monad.xyz pnpm --filter @themoss/protocol-<nft-project> test
   ```

   验证：

   - simulation 不 revert
   - `warnings` 为空
   - `effects.nftsIn` 符合预期
   - observation 正常渲染

### 4.6 接入 MCP Server

完成 Adapter 后，把新协议加入 Moss MCP server：

- 在 `packages/mcp-server/package.json` 添加依赖
- 在 `packages/mcp-server/src/server.ts` 中引入 manifest
- 在 registry 组装数组中添加新 manifest

这样 Codex 或其他 AI Agent 就能通过 Moss MCP 工具发现并调用新的 NFT mint 能力。
